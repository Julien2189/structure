SQL & MySQL

Des premières requêtes SELECT aux jointures complexes, en passant par les index, les transactions et l'intégration avec Node.js et PHP. La base de données de 90% des projets web.

🗄️ 1 — Pourquoi SQL ?

SQL (Structured Query Language) est le langage qui permet de parler à une base de données relationnelle. MySQL, PostgreSQL, SQLite, MariaDB — tous comprennent SQL. C'est le standard depuis 50 ans, et ça ne changera pas de sitôt.

Sans base de données
  • Données perdues à chaque redémarrage
  • Impossible de chercher/filtrer efficacement
  • Pas de relations entre les données
  • Un seul utilisateur à la fois
Avec MySQL
  • Persistance garantie sur disque
  • Recherche et tri en millisecondes sur des millions de lignes
  • Relations entre tables (users ↔ commandes ↔ produits)
  • Des milliers de connexions simultanées

Créer une table complète

CREATE TABLE — structure réalisteSQL
CREATE TABLE utilisateurs (
  -- Clé primaire auto-incrémentée
  id          INT UNSIGNED NOT NULL AUTO_INCREMENT,
  nom         VARCHAR(100) NOT NULL,
  email       VARCHAR(255) NOT NULL UNIQUE,
  mot_de_passe VARCHAR(255) NOT NULL,
  role        ENUM('admin', 'user', 'guest') NOT NULL DEFAULT 'user',
  actif       BOOLEAN NOT NULL DEFAULT TRUE,
  cree_le     TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  modifie_le  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  PRIMARY KEY (id),
  INDEX idx_email (email)   -- index pour recherches rapides par email
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Table commandes avec clé étrangère
CREATE TABLE commandes (
  id           INT UNSIGNED NOT NULL AUTO_INCREMENT,
  utilisateur_id INT UNSIGNED NOT NULL,
  total        DECIMAL(10,2) NOT NULL,
  statut       ENUM('en_attente','confirmee','livree','annulee') DEFAULT 'en_attente',
  cree_le      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

  PRIMARY KEY (id),
  -- Clé étrangère : utilisateur_id → utilisateurs.id
  FOREIGN KEY (utilisateur_id) REFERENCES utilisateurs(id)
    ON DELETE CASCADE   -- si user supprimé → ses commandes aussi
    ON UPDATE CASCADE
) ENGINE=InnoDB;

-- Modifier une table existante
ALTER TABLE utilisateurs ADD COLUMN avatar_url VARCHAR(500) NULL;
ALTER TABLE utilisateurs MODIFY COLUMN nom VARCHAR(150) NOT NULL;
ALTER TABLE utilisateurs DROP COLUMN avatar_url;
⚠️ ON DELETE CASCADE — attention

ON DELETE CASCADE supprime automatiquement les enregistrements liés. Si tu supprimes un utilisateur, toutes ses commandes disparaissent aussi. C'est parfois voulu (commentaires, sessions) et parfois catastrophique (données comptables). L'alternative est ON DELETE SET NULL ou RESTRICT (bloque la suppression).

🔍 3 — SELECT — Lire des Données

Le SELECT est la requête la plus utilisée. Son ordre d'exécution logique est différent de l'ordre d'écriture — comprendre ça évite beaucoup de confusion.

Ordre d'exécution logique d'un SELECT
1. FROM & JOIN  — quelles tables ?
2. WHERE          — filtrer les lignes
3. GROUP BY     — regrouper
4. HAVING       — filtrer les groupes
5. SELECT       — quelles colonnes ?
6. ORDER BY     — trier
7. LIMIT / OFFSET — paginer
L'ordre d'écriture SQL ne reflète pas l'ordre d'exécution. WHERE s'exécute avant SELECT — c'est pourquoi on ne peut pas utiliser un alias de SELECT dans un WHERE.

🎯 4 — WHERE — Filtrer les Données

Le WHERE est la clause la plus puissante pour sélectionner exactement les lignes dont tu as besoin.

WHERE — tous les opérateursSQL
-- Comparaisons simples
SELECT * FROM produits WHERE prix > 50;
SELECT * FROM utilisateurs WHERE role = 'admin';
SELECT * FROM utilisateurs WHERE role != 'guest';

-- Combinaisons logiques
SELECT * FROM produits
WHERE prix >= 20 AND prix <= 60;

SELECT * FROM utilisateurs
WHERE role = 'admin' OR role = 'user';

-- BETWEEN (inclusif des deux bornes)
SELECT * FROM produits
WHERE prix BETWEEN 20 AND 60;

-- IN — liste de valeurs
SELECT * FROM utilisateurs
WHERE role IN ('admin', 'user');

-- NOT IN
SELECT * FROM commandes
WHERE statut NOT IN ('annulee');

-- LIKE — recherche par motif
-- % = n'importe quelle suite de caractères
-- _ = exactement 1 caractère
SELECT * FROM utilisateurs
WHERE email LIKE '%@gmail.com';       -- se termine par @gmail.com

SELECT * FROM produits
WHERE nom LIKE '%shirt%';              -- contient "shirt"

-- IS NULL / IS NOT NULL
SELECT * FROM utilisateurs WHERE avatar_url IS NULL;
SELECT * FROM utilisateurs WHERE avatar_url IS NOT NULL;

-- Combiner avec parenthèses
SELECT * FROM commandes
WHERE (statut = 'livree' OR statut = 'confirmee')
  AND total > 50;

-- Recherche full-text (sur colonne indexée FULLTEXT)
SELECT * FROM articles
WHERE MATCH(titre, contenu) AGAINST('intelligence artificielle' IN BOOLEAN MODE);
⚠️ NULL — le cas particulier

En SQL, NULL signifie "valeur inconnue". NULL = NULL vaut NULL (pas TRUE) ! C'est pourquoi on doit utiliser IS NULL et jamais = NULL. De même, NULL != 'valeur' vaut NULL, pas TRUE — ce qui signifie que les lignes avec NULL sont exclues des WHERE classiques.

🔗 5 — Jointures — Relier les Tables

Les jointures permettent de combiner des données de plusieurs tables en une seule requête. C'est la fonctionnalité la plus puissante de SQL et la plus mal comprise.

Les 4 types de JOIN

INNER JOIN
A
B
Seulement les lignes qui ont une correspondance dans les deux tables. Le plus utilisé.
LEFT JOIN
A
B
Toutes les lignes de la table de gauche + correspondances de droite. NULL si pas de match à droite.
RIGHT JOIN
A
B
Toutes les lignes de droite + correspondances de gauche. Moins utilisé — souvent remplaçable par LEFT JOIN inversé.
FULL OUTER JOIN
A
B
Toutes les lignes des deux tables. Non natif en MySQL — simulé avec UNION de LEFT + RIGHT JOIN.
JOIN — exemples concrets sur la boutiqueSQL
-- INNER JOIN : commandes avec le nom du client
-- (seulement les clients qui ont commandé)
SELECT
  c.id          AS commande_id,
  u.nom         AS client,
  u.email,
  c.total,
  c.statut
FROM commandes c
INNER JOIN utilisateurs u ON c.utilisateur_id = u.id
ORDER BY c.cree_le DESC;

-- LEFT JOIN : TOUS les utilisateurs,
-- avec leurs commandes s'ils en ont
SELECT
  u.nom,
  COUNT(c.id)    AS nb_commandes,
  COALESCE(SUM(c.total), 0) AS total_depense
FROM utilisateurs u
LEFT JOIN commandes c ON u.id = c.utilisateur_id
GROUP BY u.id, u.nom
ORDER BY total_depense DESC;
-- David (0 commande) apparaît avec nb=0, total=0

-- LEFT JOIN pour trouver les clients SANS commande
SELECT u.nom, u.email
FROM utilisateurs u
LEFT JOIN commandes c ON u.id = c.utilisateur_id
WHERE c.id IS NULL;  -- le truc classique !

-- JOIN triple : commandes + clients + lignes de commande + produits
SELECT
  u.nom      AS client,
  c.id       AS commande,
  p.nom      AS produit,
  lc.quantite,
  lc.quantite * p.prix AS sous_total
FROM commandes c
JOIN utilisateurs      u  ON c.utilisateur_id  = u.id
JOIN lignes_commande   lc ON lc.commande_id     = c.id
JOIN produits          p  ON lc.produit_id      = p.id
WHERE c.statut  'annulee'
ORDER BY c.id, p.nom;

📊 6 — Fonctions d'Agrégat & GROUP BY

Les fonctions d'agrégat calculent une valeur à partir de plusieurs lignes : compter, sommer, faire des moyennes. Combinées avec GROUP BY, elles produisent des statistiques puissantes.

Agrégats — COUNT, SUM, AVG, MIN, MAXSQL
-- Compter les utilisateurs
SELECT COUNT(*) AS total_users FROM utilisateurs;

-- COUNT(colonne) ignore les NULL
SELECT COUNT(avatar_url) AS avec_avatar FROM utilisateurs;

-- Statistiques sur les prix
SELECT
  COUNT(*) AS nb_produits,
  AVG(prix)  AS prix_moyen,
  MIN(prix)  AS moins_cher,
  MAX(prix)  AS plus_cher,
  SUM(stock) AS total_stock
FROM produits;

-- GROUP BY : statistiques par groupe
SELECT
  role,
  COUNT(*) AS nb_utilisateurs
FROM utilisateurs
GROUP BY role
ORDER BY nb_utilisateurs DESC;

-- Chiffre d'affaires par statut de commande
SELECT
  statut,
  COUNT(*) AS nb,
  SUM(total) AS ca_total,
  ROUND(AVG(total), 2) AS panier_moyen
FROM commandes
GROUP BY statut;

-- HAVING : filtrer les groupes (≠ WHERE qui filtre les lignes)
SELECT
  utilisateur_id,
  COUNT(*) AS nb_commandes,
  SUM(total) AS total_depense
FROM commandes
WHERE statut != 'annulee'      -- filtrer avant groupement
GROUP BY utilisateur_id
HAVING SUM(total) > 100        -- filtrer après groupement
ORDER BY total_depense DESC;

Fonctions SQL utiles

Fonctions de chaînes, dates et conditionnellesSQL
-- Chaînes
SELECT
  UPPER(nom),          -- ALICE
  LOWER(email),        -- alice@mail.fr
  LENGTH(nom),         -- 5
  TRIM(nom),           -- supprime espaces début/fin
  CONCAT(nom, ' ', email),
  SUBSTRING(nom, 1, 3)  -- 3 premiers caractères
FROM utilisateurs;

-- Dates
SELECT
  NOW(),                         -- date + heure actuelle
  CURDATE(),                     -- date du jour
  DATE_FORMAT(cree_le, '%d/%m/%Y'),
  DATEDIFF(NOW(), cree_le) AS jours_anciennete,
  YEAR(cree_le), MONTH(cree_le)
FROM utilisateurs;

-- Commandes du dernier mois
SELECT * FROM commandes
WHERE cree_le >= DATE_SUB(NOW(), INTERVAL 30 DAY);

-- CASE WHEN — condition dans le SELECT
SELECT
  nom,
  prix,
  CASE
    WHEN prix < 25  THEN 'Économique'
    WHEN prix < 70  THEN 'Milieu de gamme'
    ELSE 'Premium'
  END AS categorie_prix,
  COALESCE(description, 'Pas de description') AS desc_safe
FROM produits;

UPDATE & DELETE
Modifier et supprimerSQL
-- UPDATE — toujours avec WHERE !
UPDATE utilisateurs
SET
  nom = 'Julien L.',
  role = 'admin'
WHERE id = 5;

-- UPDATE avec calcul
UPDATE produits
SET prix = prix * 0.90  -- -10%
WHERE stock > 100;

-- DELETE — toujours avec WHERE !
DELETE FROM commandes
WHERE statut = 'annulee'
  AND cree_le < '2023-01-01';

-- TRUNCATE — vider une table entière
-- (plus rapide que DELETE sans WHERE)
TRUNCATE TABLE sessions;
⚠️ UPDATE et DELETE sans WHERE = catastrophe

Un UPDATE utilisateurs SET role = 'guest' sans WHERE met tous les utilisateurs en guest. Un DELETE FROM commandes sans WHERE supprime toute la table. Bonne pratique : teste toujours ta condition avec un SELECT d'abord, puis remplace SELECT par UPDATE/DELETE.

Un bon schéma évite la redondance, les incohérences et les problèmes de performance. La normalisation est l'ensemble des règles pour y parvenir.

💡 Règles de normalisation essentielles

Ne jamais dupliquer les données — si le nom d'un produit doit changer, il ne doit changer qu'à un seul endroit.
Stocker le prix unitaire dans lignes_commande — pas dans produits uniquement. Le prix peut changer, mais la commande historique doit garder le prix au moment de l'achat.
Tables de jointure pour N-N — un produit peut être dans plusieurs commandes, une commande peut avoir plusieurs produits → table lignes_commande.
Clés étrangères obligatoires — elles garantissent l'intégrité référentielle (pas de commande orpheline).

⚡ 9 — Index — Accélérer les Requêtes

Un index est une structure de données auxiliaire (B-Tree) qui permet à MySQL de trouver des lignes sans parcourir toute la table. Sans index sur 1 million de lignes, une recherche par email peut prendre plusieurs secondes. Avec, quelques microsecondes.

Sans index — full table scan

MySQL lit chaque ligne une par une pour trouver celle qui correspond. Sur 1M lignes → 1M lectures.

⏱️ ~800ms sur 1M lignes
Avec index — B-Tree lookup

MySQL descend dans l'arbre B-Tree — log₂(1M) ≈ 20 lectures seulement pour trouver la valeur.

⚡ ~1ms sur 1M lignes
Créer, analyser et gérer les indexSQL
-- Créer un index simple
CREATE INDEX idx_email     ON utilisateurs(email);
CREATE INDEX idx_statut    ON commandes(statut);
CREATE INDEX idx_cree_le  ON commandes(cree_le);

-- Index UNIQUE (empêche les doublons + indexe)
CREATE UNIQUE INDEX idx_email_unique ON utilisateurs(email);

-- Index composite (plusieurs colonnes)
-- Utile pour : WHERE statut = 'livree' AND utilisateur_id = 5
CREATE INDEX idx_user_statut ON commandes(utilisateur_id, statut);

-- EXPLAIN — voir comment MySQL exécute une requête
EXPLAIN SELECT * FROM utilisateurs WHERE email = 'alice@mail.fr';
-- Colonnes importantes :
-- type = 'ref' ou 'eq_ref' → index utilisé ✅
-- type = 'ALL' → full scan, pas d'index ❌
-- rows = nombre estimé de lignes lues
-- key = index utilisé

-- Voir les index d'une table
SHOW INDEX FROM utilisateurs;

-- Supprimer un index
DROP INDEX idx_email ON utilisateurs;
💡 Règles des index

Indexer obligatoirement : les colonnes dans WHERE, JOIN ON, ORDER BY fréquents. Les clés étrangères.
Ne pas sur-indexer : chaque index ralentit les INSERT/UPDATE/DELETE (l'index doit aussi être mis à jour). Tables avec peu de lignes n'ont pas besoin d'index.
Index composite : l'ordre compte — (user_id, statut) accélère les recherches sur user_id seul ou user_id + statut, mais pas sur statut seul.

🔒 10 — Transactions & ACID

Une transaction est un groupe d'opérations SQL qui réussissent toutes ensemble ou échouent toutes ensemble. Indispensable pour les opérations critiques (paiement, transfert, stock).

Propriétés ACID — garanties d'une transaction
Atomicité    — tout ou rien. Pas de demi-transaction.
Cohérence    — la DB reste dans un état valide avant et après.
Isolation     — les transactions concurrentes ne se perturbent pas.
Durabilité   — une transaction committée est permanente, même si le serveur crash.
Transaction — achat avec décrémentation de stockSQL
-- Scenario : Bob achète 2 T-shirts

START TRANSACTION;

-- 1. Vérifier le stock disponible
SELECT stock FROM produits
WHERE id = 1
FOR UPDATE;  -- verrouille la ligne le temps de la transaction

-- Si stock insuffisant → ROLLBACK dans l'application

-- 2. Créer la commande
INSERT INTO commandes (utilisateur_id, total)
VALUES (2, 50.00);

-- 3. Ajouter la ligne de commande
INSERT INTO lignes_commande
  (commande_id, produit_id, quantite, prix_unitaire)
VALUES
  (LAST_INSERT_ID(), 1, 2, 25.00);

-- 4. Décrémenter le stock
UPDATE produits
SET stock = stock - 2
WHERE id = 1;

-- Tout s'est bien passé → valider
COMMIT;

-- En cas d'erreur → annuler tout
-- ROLLBACK;
-- La commande ET la décrémentation sont annulées

🔌 11 — Node.js + MySQL2

Intégrer MySQL dans une API Express. Le package mysql2 est le plus performant et supporte les Promises et les requêtes préparées nativement.

Installation et connexionBASH
npm install mysql2 dotenv
src/db.js — pool de connexionsNODE
require('dotenv').config();
const mysql = require('mysql2/promise');

// Pool = ensemble de connexions réutilisables
// Beaucoup plus efficace qu'une connexion par requête
const pool = mysql.createPool({
  host:             process.env.DB_HOST,
  port:             process.env.DB_PORT || 3306,
  database:         process.env.DB_NAME,
  user:             process.env.DB_USER,
  password:         process.env.DB_PASSWORD,
  waitForConnections: true,
  connectionLimit:  10,   // max 10 connexions simultanées
  charset:          'utf8mb4'
});

// Tester la connexion au démarrage
(async () => {
  try {
    const conn = await pool.getConnection();
    console.log('✅ MySQL connecté');
    conn.release();
  } catch (err) {
    console.error('❌ MySQL erreur:', err.message);
    process.exit(1);
  }
})();

module.exports = pool;
src/routes/users.js — CRUD complet avec MySQLEXPRESS
const express = require('express');
const pool    = require('../db');
const router  = express.Router();

// GET /api/users — liste paginée
router.get('/', async (req, res) => {
  try {
    const page   = parseInt(req.query.page)  || 1;
    const limit  = parseInt(req.query.limit) || 20;
    const offset = (page - 1) * limit;

    // ? = paramètre préparé — protège des injections SQL
    const [rows] = await pool.execute(
      'SELECT id, nom, email, role, cree_le FROM utilisateurs ORDER BY id LIMIT ? OFFSET ?',
      [limit, offset]
    );
    const [[{ total }]] = await pool.execute(
      'SELECT COUNT(*) AS total FROM utilisateurs'
    );
    res.json({ total, page, limit, data: rows });
  } catch (err) {
    res.status(500).json({ erreur: err.message });
  }
});

// GET /api/users/:id
router.get('/:id', async (req, res) => {
  const [rows] = await pool.execute(
    'SELECT id, nom, email, role FROM utilisateurs WHERE id = ?',
    [req.params.id]
  );
  if (!rows.length) return res.status(404).json({ erreur: 'Introuvable' });
  res.json(rows[0]);
});

// POST /api/users
router.post('/', async (req, res) => {
  const { nom, email, mot_de_passe } = req.body;
  const bcrypt = require('bcryptjs');
  const hash = await bcrypt.hash(mot_de_passe, 10);

  const [result] = await pool.execute(
    'INSERT INTO utilisateurs (nom, email, mot_de_passe) VALUES (?, ?, ?)',
    [nom, email, hash]
  );
  res.status(201).json({ id: result.insertId, nom, email });
});

// Transaction — achat
router.post('/acheter', async (req, res) => {
  const conn = await pool.getConnection();
  try {
    await conn.beginTransaction();

    const { userId, produitId, qte } = req.body;

    // Vérifier stock
    const [[prod]] = await conn.execute(
      'SELECT prix, stock FROM produits WHERE id = ? FOR UPDATE',
      [produitId]
    );
    if (prod.stock < qte) throw new Error('Stock insuffisant');

    // Créer commande
    const [cmd] = await conn.execute(
      'INSERT INTO commandes (utilisateur_id, total) VALUES (?, ?)',
      [userId, prod.prix * qte]
    );
    await conn.execute(
      'INSERT INTO lignes_commande (commande_id, produit_id, quantite, prix_unitaire) VALUES (?,?,?,?)',
      [cmd.insertId, produitId, qte, prod.prix]
    );
    await conn.execute(
      'UPDATE produits SET stock = stock - ? WHERE id = ?',
      [qte, produitId]
    );

    await conn.commit();
    res.status(201).json({ commandeId: cmd.insertId });
  } catch (err) {
    await conn.rollback();
    res.status(400).json({ erreur: err.message });
  } finally {
    conn.release(); // TOUJOURS libérer la connexion
  }
});

module.exports = router;

🛡️ 12 — Sécurité SQL — Injections & Bonnes Pratiques

L'injection SQL est la faille de sécurité web la plus commune et la plus dévastatrice. Une seule requête mal construite peut exposer toute ta base de données.

❌ Code vulnérable
JAMAIS faire çaDANGER
// L'attaquant envoie :
// email = "' OR '1'='1"
// La requête devient :
// SELECT * FROM utilisateurs
// WHERE email = '' OR '1'='1'
// → retourne TOUS les utilisateurs !

// Encore pire :
// email = "'; DROP TABLE utilisateurs; --"

const query = `SELECT * FROM utilisateurs
  WHERE email = '${req.body.email}'`;
// CONCATÉNATION DIRECTE = INJECTION SQL !
await conn.query(query); // ☠️
✅ Code sécurisé
Requêtes préparéesSÉCURISÉ
// Les ? sont des paramètres liés
// MySQL échappe automatiquement

✅ Checklist sécurité SQL

✅ Toujours des requêtes préparées — jamais de concaténation de variables utilisateur dans du SQL
✅ Valider les données côté serveur avant d'insérer (longueur, format, type)
✅ Principe du moindre privilège — le user MySQL de l'app n'a que SELECT/INSERT/UPDATE/DELETE, jamais DROP/CREATE/GRANT
✅ Ne jamais exposer les erreurs MySQL en production — loguer côté serveur, répondre "erreur serveur" au client
✅ Mots de passe hashés avec bcrypt — jamais en clair en base
